## 长上下文处理 in Agent

```
from langchain_openai import ChatOpenAI
from langchain_core.messages import SystemMessage, HumanMessage

llm = ChatOpenAI(model="gpt‐4o", temperature=0.1)
message_history = [HumanMessage(content=task_prompt), ...]
response = llm.invoke(message_history)
```

我们需要从多个方面进行搜索：一是LangChain中处理长对话历史的最佳实践；二是Claude Code处理大上下文的机制；三是一些通用的长上下文处理策略。


### LangChain提供的解决方案：
1. 滑动窗口 (Sliding Window)
2. 对话摘要 (Conversation Summary)
3. 智能裁剪 (Semantic / Token Trimming)
   1. eg. call trim_messages()
   2. eg. LLM do 语义压缩 (Semantic Compression)

### Claude Code 如何处理超大上下文？
核心机制如下：
1. Auto-Compaction (自动压缩)
2. 分层压缩策略 (Three-Layer Compression)
3. 提示缓存 (Prompt Caching)
   1. 缓存在Anthropic API server端。

### 🧩 主流长上下文处理策略

1. 全量记忆
   1. push all to message_history
2. 滑动窗口
   1. last k msg
3. 摘要/压缩
   1. LLM 对 message_history 摘要 并 替换
   2. eg. langchain summarizationMiddleware()
4. 相关性过滤
   1. 为每一条历史信息打分，只保留重要的内容，丢弃次要信息。
5. 向量数据库 / 检索增强生成
   1. 将历史消息向量化后存入向量数据库，每次交互时检索最相关的 K 条消息注入上下文
   2. 适用场景：个性化助手、跨会话记忆、知识库问答
6. 分层记忆
   1. 结合短期记忆（滑动窗口）和长期记忆（向量数据库），实现类似人脑的分层管理。
   2. 适用场景：个性化助理、跨天日程管理、企业级客服系统。
7. 类 OS 内存管理（如 MemGPT）
   1. 模拟操作系统的内存管理机制，将近期对话视为 RAM，历史信息视为 Disk，按需进行 Page In/Page Out。
   2. 适用场景：需要低延迟且具备长期高容量记忆的对话系统，如跨天任务管理、长期个性化助手。
8. 提示缓存 (Prompt Caching)
   1. 利用部分模型提供的缓存能力，暂存已处理过的上下文，避免重复计算。
9. 递归摘要 / 树状索引
   1.  将长文档或对话历史分层切块，递归生成摘要，最终构建一棵摘要树（如 RAPTOR 模型）。
   2.  适用场景：超长文档分析、复杂推理问答、企业级知识库系统。
10. 子代理与任务委派
    1.  分布式处理长上下文

## 主流 LLM 的 Context Window 工作原理

### 1. 上下文窗口的核心概念

**上下文窗口（Context Window）** 是指模型在一次前向计算中能够“看见”的最大 Token 数量。它决定了模型在生成下一个 Token 时可以参考多少历史信息。

- **Token**：文本的最小处理单元，可能是单词片段、标点或特殊符号。
- **窗口长度限制**：例如 GPT-4 Turbo 为 128k tokens，Claude 3 为 200k tokens。

### 2. 原理简述（Transformer 解码器）

LLM 基于自注意力机制（Self-Attention）工作。每生成一个新 token，模型都会计算当前位置与之前所有位置的关联权重，这个“之前所有位置”的集合大小就是上下文窗口的物理上限。

为了提升推理效率，工程实现中使用了 **KV 缓存（Key-Value Cache）**：
- 每生成一个 token，其对应的 Key 和 Value 向量会被存入缓存。
- 后续步骤只需用当前 Query 与缓存中的 K、V 进行计算，无需重新处理历史 token。

如果对话长度超过窗口上限，最早的部分将被截断，模型会“遗忘”这些内容。

### 3. 简单示例

假设模型的上下文窗口大小为 **8 tokens**，对话过程如下：
```
[中国, 的, 首都, 是, 哪里, ？] // 6 tokens
[中国, 的, 首都, 是, 北京, 。] // 6 tokens
[那里, 有, 什么, 著名, 景点, ？] // 6 tokens
[北京, 有, 故宫, 、, 长城, 等, 。] // 7 tokens
```

当总 token 数超过 8 时，模型必须截断最早的内容（例如删除第一轮的用户提问）。如果被截断的部分包含关键信息，回答质量将受到影响。

## LLM API 服务端如何为不同连接维护 Context Window / Cache？

### 1. API 服务的无状态特性

大部分 LLM API（如 OpenAI、Anthropic）采用 **无状态（Stateless）** 设计。每次 HTTP 请求都是独立的，服务端不会自动记住上一次对话内容。

- 调用方需要在每次请求中 **完整传递对话历史**（即 `messages` 数组）。
- 服务端收到后，会将历史消息拼接、分词，并检查总 token 数是否超出模型窗口上限。
- 超出上限时，部分服务会返回错误，部分会自动截断早期消息（规则因厂商而异）。

### 2. 服务端内部的跨连接缓存复用优化

尽管连接之间无状态，推理引擎底层仍通过多种技术实现高效复用：

| 技术 | 作用 |
| :--- | :--- |
| **前缀缓存（Prefix Caching）** | 对于相同的系统提示或长文档前缀，缓存其 KV 值，后续请求可直接复用，避免重复计算。 |
| **连续批处理（Continuous Batching）** | 将不同用户的请求动态合并为批次执行，在 GPU 上并行计算，减少空闲时间。 |
| **会话级缓存（Session-level Cache）** | 部分平台（如 Assistants API）在服务端存储对话历史，用户只需发送新消息，但底层推理仍需传递完整窗口。 |

### 3. API 语境下的“上下文窗口”与 LLM 本身概念的关系

| 比较维度 | LLM 本身的 Context Window | API 调用中的 Context Window |
| :--- | :--- | :--- |
| **定义** | 模型单次推理能处理的最大 token 数 | 单次请求中 `messages` 总 token 数的上限 |
| **维护方** | 模型架构决定，推理时通过 KV 缓存实现 | 调用方负责控制传入内容的长度 |
| **跨请求保留** | 同一推理会话内保留（生成下一个 token 时复用缓存） | 不同 HTTP 请求间 **不保留**，需每次发送完整历史 |

**结论**：两者本质相同，区别在于职责边界——API 服务不跨请求维护上下文，而底层优化（如前缀缓存）则是对开发者透明的性能增强。

## LLM API服务端内部优化“海量连接”下缓存复用典型策略

### 核心优化策略一览

| 技术策略 | 解决的核心问题 | 主要收益 |
| :--- | :--- | :--- |
| **PagedAttention（分页注意力）** | 传统 KV Cache 内存浪费与碎片化严重 | 内存利用率提升 3-5 倍，减少 78% 的内存碎片 |
| **Continuous Batching（连续批处理）** | 传统静态批处理导致的“气泡效应” | GPU 利用率从 62% 提升至 89% 以上 |
| **Prefix Caching（前缀缓存）** | 重复计算相同前缀（系统提示、长文档） | 显著减少 Prefill 时间，降低首 Token 延迟（TTFT） |
| **Prefix-Aware Routing（前缀感知路由）** | 多节点集群中前缀缓存的命中率低 | 确保同类请求路由至相同节点，提高缓存复用率 |

### 1. 基石：PagedAttention 与 KV Cache 内存管理

高并发场景下，服务端需同时维护成千上万个对话的 KV Cache，传统方式（预留连续显存）会造成严重的内存碎片和浪费。**PagedAttention** 借鉴操作系统虚拟内存思想：

- **分块管理**：将显存划分为固定大小的“页”（例如 16 或 32 token/页）。
- **按需分配**：请求按需获取物理页，通过页表映射形成逻辑序列。
- **消除碎片**：页面回收后可立即分配给其他请求，极大提升显存利用率（测试提升 3-5 倍），使单卡支持更多并发连接。

### 2. 调度核心：连续批处理（Continuous Batching）

传统静态批处理存在 **气泡效应**：批次中短请求需等待长请求完成，期间 GPU 空闲。连续批处理通过 **迭代级调度** 解决此问题：

- **持续运转**：GPU 每次迭代（生成一个 token）后立即检查是否有请求完成。
- **动态插队**：完成的请求位置被新请求无缝填补，保持 GPU 满载。
- **效果**：GPU 利用率从传统 60% 左右提升至 90% 以上，成为高并发支撑的关键。

### 3. 跨连接优化：前缀缓存（Prefix Caching）

大量跨连接请求共享相同前缀（系统提示、长文档、历史对话）。前缀缓存允许服务端“记住”并复用这些公共前缀的 KV Cache。

- **自动前缀缓存（APC）**：vLLM 等引擎在首次计算时缓存完整 KV，后续相同前缀请求直接复用，仅计算新后缀部分。
- **收益**：
  - 大幅降低首 Token 延迟（TTFT），节省重复的预填充（Prefill）时间。
  - 节省显存：公共前缀在显存中仅存一份，多请求共享。

### 4. 集群级优化：前缀感知路由（Prefix-Aware Routing）

在多 GPU 节点集群中，为确保相同前缀的请求命中已缓存节点，引入前缀感知路由：

- **机制**：网关分析请求前缀哈希或语义，将请求“粘性”转发至缓存了该前缀的后端实例。
- **效果**：避免随机负载均衡导致的缓存不命中与重复计算，在 RAG、Agent 系统中尤为重要。

### 总结

服务端高效处理海量连接的背后，是一套精密协作的技术体系：
- **PagedAttention**：内存管家，提升显存容纳能力。
- **连续批处理**：调度指挥官，保持 GPU 持续饱和。
- **前缀缓存**：记忆大师，避免重复劳动。
- **前缀感知路由**：集群导航员，精准利用已有缓存。

这些技术共同实现了低成本、高并发、低延迟的 LLM API 服务。
